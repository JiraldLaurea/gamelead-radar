Implement the tasks in `docs/plans/tasks/` according to the dependency graph. The output is working source code, unit tests, API tests, and commits.

## Stage 0: Pre-Execution Environment Check

Run once before implementation begins.

```text
1. Confirm project directory.
2. Confirm package.json and tsconfig.json exist.
3. Confirm dependencies are installed. If node_modules is missing, run pnpm install.
4. Confirm dev server starts without errors.
5. Confirm existing tests pass.
6. Confirm Git working tree is clean.
7. Confirm DB is reachable.
8. Confirm migrations are up to date.
9. Confirm required environment variables exist.
10. Confirm no conflicting process occupies required ports.
```

### Commands

```bash
pnpm install
pnpm typecheck
pnpm lint
pnpm test
pnpm build
git status
```

If the project uses Docker:

```bash
docker compose up -d
pnpm db:status
```

If the working tree has uncommitted changes, report them before proceeding. Do not overwrite working code unnecessarily.

---

## Stage 1: Build the Task Queue

1. Read every task file under `docs/plans/tasks/`.
2. Read `_dependency-graph.md`.
3. Group tasks into parallel execution groups.
4. Validate BE-FE CRUD correspondence before confirming the queue.
5. Show the planned execution order.

Example:

```text
Group 1: INF-DB-1, INF-CI-1
Group 2: BE-AUTH-1, BE-AUTH-2
Group 3: FE-AUTH-1, FE-AUTH-2
Group 4: FS-BOOKING-1
```

### Execution Order Principle

```text
1. INF tasks first
2. DB migration tasks before backend API tasks
3. Backend API tasks before frontend API integration tasks
4. Frontend UI tasks after supporting APIs exist
5. Full-stack tasks after FE and BE foundations are ready
6. Test/security tasks after implementation is available
```

### Prohibited

- Do not make frontend call APIs that are not implemented.
- Do not create duplicate fetch wrappers or stores.
- Do not change field names outside the documented naming strategy.
- Do not bypass validation defined in `api-contract.md`.
- Do not skip tests required by the task file.

---

## Stage 2: Single Task Execution Protocol

Every task must follow this sequence.

---

### Step 1: Load Context

Before writing code, read:

1. The task file.
2. `design-system.md`.
3. The module `spec.md`.
4. Relevant policy documents.
5. Relevant wireframe HTML.
6. `api-contract.md` for BE/FS tasks and FE API integration tasks.
7. `architecture.md` for middleware/auth/logging patterns.
8. `db-design.md` for DB or migration tasks.
9. Existing code in the same module.
10. Outputs from dependency tasks.
11. `ui-implementation-rules.md` for UI tasks.
12. Existing validation schemas.
13. Existing shared API helper.

### Wireframe Extraction Requirement

For frontend tasks, extract and note:

- Tailwind classes or design tokens
- component hierarchy
- labels/text
- state variations
- modals/tabs/dropdowns
- sample data structure
- role-specific UI branches

Do not implement UI by approximation when a wireframe exists.

---

### Step 2: Implement Interface Contracts First

Convert the task's interface definitions into code before implementation logic.

Examples:

```typescript
export type SuccessResponse<T> = {
  success: true;
  data: T;
};

export type ErrorResponse = {
  success: false;
  error: {
    code: string;
    message: string;
  };
};
```

Rules:

- Do not redesign types that are already specified.
- Use shared types when available.
- Do not duplicate enum definitions.
- Keep API request/response shapes aligned with `api-contract.md`.

---

### Step 3: Implement Backend Work

For backend tasks:

1. Implement or update DB schema/migration if required.
2. Implement repository functions.
3. Implement service logic.
4. Implement validation schemas.
5. Implement controller/API route.
6. Implement error handling.
7. Implement unit/API tests.
8. Run validation commands.

Required checks:

```bash
pnpm typecheck
pnpm lint
pnpm test
pnpm build
```

API verification:

```bash
curl -X POST http://localhost:3000/api/... \
  -H "Content-Type: application/json" \
  -d '{}'
```

---

### Step 4: Implement Frontend Work

For frontend tasks:

1. Implement page/route.
2. Implement components based on wireframe hierarchy.
3. Implement form validation.
4. Implement state management.
5. Integrate API with the shared API client.
6. Implement loading, error, empty, disabled, and success states.
7. Implement role-based rendering.
8. Add component/unit tests.
9. Run visual verification.

Required checks:

```bash
pnpm typecheck
pnpm lint
pnpm test
pnpm build
```

---

### Step 5: Implement Full-Stack Work

For full-stack tasks:

1. Confirm dependent DB/API tasks are complete.
2. Implement backend changes first.
3. Validate API behavior.
4. Implement frontend integration.
5. Validate end-to-end user flow.
6. Add both backend and frontend tests.

---

### Step 6: Test the Task

Minimum test requirements:

- Backend task: at least one `*.test.ts` file.
- Frontend task: at least one `*.test.tsx` file.
- Full-stack task: backend test + frontend test + integration or E2E reference.
- Security-sensitive task: validation or authorization test.

Do not mark the task complete if test files are missing.

---

### Step 7: Commit the Task

After validation passes:

```bash
git status
git add .
git commit -m "{type}: {TASK-ID} {short summary}"
```

Suggested commit types:

```text
feat, fix, test, refactor, chore, docs, ci, security
```

## Stage 3: Visual Verification Gate

For every screen in `_screen-inventory.md`:

1. Start the dev server.
2. Navigate to the screen.
3. Capture a screenshot.
4. Compare against the wireframe.
5. Verify layout, text, buttons, forms, states, and role branches.
6. Save screenshots to:

```text
docs/screenshots/{SCR-ID}.png
```

### Required Validation Categories

- layout structure
- major components
- text labels
- form behavior
- loading/error/empty states
- role-based visibility

---

## Stage 4: Full Validation After All Tasks

Run the full validation suite:

```bash
pnpm typecheck
pnpm lint
pnpm test
pnpm build
```

For API verification, confirm every endpoint from `api-contract.md` works.

For DB validation:

```bash
pnpm db:status
pnpm db:seed
```

For Git:

```bash
git status
git log --oneline -10
```

---

## Stage 5: Failure Handling

When a failure occurs:

1. Identify the exact command and error.
2. Determine whether it is caused by code, config, dependency, environment, DB, Git, or port/process conflict.
3. Fix the smallest safe scope.
4. Re-run the failing command.
5. Re-run the full validation if the fix affects shared code.
6. Record the issue and fix in `docs/logs/YYYY-MM-DD.md`.

If the same error repeats twice, try a different approach rather than repeating the same patch.

If maximum retries are exceeded, stop and report:

- error summary
- attempted fixes
- current project state
- recommended next action
