# Initial Project Setup Instruction — Step 3

Create the project skeleton based on the PRD, planning documents, and system design outputs. This project skeleton becomes the source of code patterns for task creation and development execution.

---

## Required Pre-Check and Stabilization

Before creating or modifying any code, perform a stabilization audit:

1. Confirm the project root path.
2. Confirm whether this is a new project or an existing project.
3. If the project exists, inspect current files before overwriting anything.
4. Detect duplicated configuration files, stale initialization, incomplete resets, and conflicting settings.
5. Check dependency and version conflicts.
6. Check environment-variable conflicts.
7. Check database/Git state conflicts.
8. Check port and running-process conflicts.
9. Check project-structure inconsistencies.
10. Create backups before replacing working code or configuration.

### Isolation Requirement for New Projects

Every new project must have independent infrastructure:

- separate Git repository
- separate database/project instance
- independent `.env` files
- isolated deployment settings
- project-specific configuration
- independent CI/CD secrets

Do not reuse another project's Git, DB, env, or deployment configuration unless the user explicitly requests it.

## Phase A: Project Initialization

### A-1. Decide Project Structure

Use `architecture.md` ADR-02 to choose one of the following:

#### Case 1: Next.js Full-Stack with API Routes

```bash
pnpm create next-app@latest {project-name} --typescript --tailwind --eslint --app --src-dir
cd {project-name}
```

#### Case 2: Next.js Frontend + Separate Backend Server

```bash
mkdir {project-name} && cd {project-name}
pnpm init
mkdir -p apps/web apps/api packages/shared
```

Create `pnpm-workspace.yaml`:

```yaml
packages:
  - "apps/*"
  - "packages/*"
```

### A-2. Git Initialization

```bash
git init
git branch -M main
```

Create a clean `.gitignore`:

```text
node_modules
.next
dist
.env
.env.*
!.env.example
coverage
playwright-report
test-results
.DS_Store
```

### A-3. Common Configuration

Create or confirm:

```text
tsconfig.json
.eslintrc.*
.prettierrc
.env.example
README.md
```

TypeScript must use strict mode.

### Validation

```bash
pnpm typecheck
pnpm lint
```

## Phase B: Frontend Setup

### B-1. Tailwind and Design Tokens

Reflect `docs/plans/design-system.md` in:

```text
tailwind.config.ts
src/app/globals.css
```

Use CSS variables for key tokens:

```css
:root {
  --color-primary: #3B82F6;
  --color-danger: #EF4444;
  --color-success: #22C55E;
  --color-warning: #F59E0B;
}
```

### B-2. Core Dependencies

```bash
pnpm add zustand @tanstack/react-query react-hook-form zod @hookform/resolvers lucide-react
pnpm add -D vitest @testing-library/react @testing-library/jest-dom jsdom @playwright/test
pnpm exec playwright install
```

### B-3. Providers

Create:

```text
src/app/providers.tsx
```

Include:

- React Query `QueryClientProvider`
- global auth/session provider if required
- toast/notification provider if required

Wrap it in `src/app/layout.tsx`.

### B-4. Store Boilerplate

Create auth store when authentication exists:

```text
src/stores/auth-store.ts
```

Include:

- user
- token/session
- `isAuthenticated`
- login/logout actions

### B-5. API Client

Create:

```text
src/lib/api-client.ts
```

Requirements:

- base URL from environment variables
- automatic auth header attachment
- common error parsing
- typed `SuccessResponse<T>` and `ErrorResponse`
- no duplicate fetch wrappers

### Validation

```bash
pnpm typecheck
pnpm lint
pnpm dev
```

## Phase C: Backend Setup

Follow ADR-02.

### Case 1: Next.js API Routes

Create structure:

```text
src/app/api/
src/server/services/
src/server/repositories/
src/server/middleware/
src/server/validators/
src/server/errors/
```

### Case 2: Separate Backend Server

Create structure:

```text
apps/api/src/
├── routes/
├── controllers/
├── services/
├── repositories/
├── middleware/
├── validators/
└── errors/
```

### Backend Requirements

- Router/controller/service/repository separation.
- Zod validation for request input.
- Common error response format from `api-contract.md`.
- Auth middleware according to `architecture.md`.
- Role-based authorization middleware.
- Request logging middleware.
- Health endpoint:

```text
GET /api/health
```

### Validation

```bash
pnpm typecheck
pnpm lint
pnpm build
```

For monorepo:

```bash
pnpm --filter api typecheck
pnpm --filter api lint
pnpm --filter api build
```

## Phase D: Database Setup

### D-1. ORM Selection

Use ADR-03 from `architecture.md`.

- Prisma: use `prisma/schema.prisma`
- Drizzle: use `src/db/schema.ts` or `apps/api/src/db/schema.ts`

### D-2. Local DB

Create `docker-compose.yml` if local DB is required.

Example PostgreSQL:

```yaml
services:
  db:
    image: postgres:16
    ports:
      - "5432:5432"
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: app
      POSTGRES_DB: app_dev
    volumes:
      - postgres_data:/var/lib/postgresql/data
volumes:
  postgres_data:
```

### D-3. Migration and Seed

Create:

```text
prisma/migrations/ or drizzle migrations
prisma/seed.ts or src/db/seed.ts
```

Seed data must match PRD roles, required enums, and minimum data for E2E tests.

### Validation

```bash
docker compose up -d
pnpm db:migrate
pnpm db:seed
pnpm db:status
```

## Phase F: CI/CD Setup

Create:

```text
.github/workflows/ci.yml
```

Minimum CI steps:

```yaml
name: CI
on:
  pull_request:
  push:
    branches: [main]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm typecheck
      - run: pnpm lint
      - run: pnpm test
      - run: pnpm build
```

Add DB service containers when backend tests require DB.
