# Full Development Pipeline — Full-Stack Project Standard

This document defines the complete workflow for a mid-sized full-stack project covering frontend, backend, database, testing, and deployment readiness. Each step uses the previous step's output as its input. Do not skip steps unless the missing outputs have already been created and validated.

> File numbering is role-based and may differ from the pipeline step number.  
> Example: Step 0 uses `1. PRD Writing.md`; Step 1 uses `2. Planning Documents.md`; Step 2 uses `3. System Design.md`.

---

## Pipeline Overview

```text
[Idea / Requirements] provided by the user
  ↓
Step 0: PRD Writing
  Output: PRD.md
  ↓
Step 1: Planning Document Generation
  Output: spec.md, policy.md, wireframe.html, design-system.md, ui-implementation-rules.md
  ↓
Step 2: System Design
  Output: architecture.md, db-design.md, api-contract.md
  ↓
Step 3: Initial Project Setup
  Output: project skeleton, DB schema, CI pipeline, shared configuration
  ↓
Step 4: Detailed Development Task Creation
  Output: docs/plans/tasks/*.md, _dependency-graph.md, _traceability-matrix.md
  ↓
Step 5: Development Execution
  Output: implemented source code, unit tests, API tests, commits
  ↓
Step 6: Testing
  Output: integration tests, E2E tests, security checks, optional performance tests
  ↓
Step 7: Deployment Preparation
  Output: deployment configuration, monitoring, rollback plan, runbook
```

---

## Step 0: PRD Writing

**Instruction file:** `1. PRD Writing.md`  
**Input:** idea, requirements, reference materials  
**Output:** `PRD.md` in the project root

### Completion Criteria

- All 16 required PRD sections exist.
- Self-validation passes in 4 categories: completeness, consistency, specificity, and downstream compatibility.
- Every enum has a label and clear allowed values.
- Every stateful entity has a state-transition table.
- Every policy maps to one or more error codes.
- Deployment environment requirements are specified, including infrastructure and hosting assumptions.

### Before Moving to Step 1

Ask the user to review the PRD, especially:

- module boundaries
- data model
- policy numbers and thresholds
- API endpoint list
- error codes
- deployment environment assumptions

Apply revisions before starting Step 1.

## Step 1: Planning Document Generation

**Instruction file:** `2. Planning Documents.md`  
**Input:** `PRD.md` from Step 0  
**Output:**

```text
docs/plans/
├── {module}/spec.md
├── policies/{policy}.md
├── wireframes/{module}.html
├── wireframes/Storyboard.html                    optional
├── wireframes/screens/{flow}_{n}_{screen}.png     optional
├── design-system.md
└── ui-implementation-rules.md
```

### Completion Criteria

- Every module has a `spec.md` and, if it has UI, a matching `wireframe.html`.
- Every policy domain has a `policy.md` file.
- Cross-validation passes across PRD, spec, policy, and wireframe outputs.
- Every wireframe screen section has a unique `SCR-...` ID.
- Every policy rule has a unique ID.
- The API section in each spec matches PRD Section 9 exactly.
- If a storyboard is generated, screen images map 1:1 to storyboard steps.

### Before Moving to Step 2

Ask the user to review the planning documents and confirm the module structure, screen flow, policy logic, and design-system direction.

---

## Step 2: System Design

**Instruction file:** `3. System Design.md`  
**Input:** PRD + Step 1 planning outputs  
**Output:**

```text
docs/architecture/
├── architecture.md
├── db-design.md
└── api-contract.md
```

### Completion Criteria

- Architecture decisions are documented with ADRs.
- Every entity has an ERD, table definition, indexes, and query patterns.
- Every API endpoint has request, response, and error contracts.
- Authentication and authorization architecture is defined.
- Cross-cutting concerns are designed: logging, error handling, caching, validation, rate limiting, and monitoring.

### Before Moving to Step 3

Ask the user to review the architecture, especially DB design, API contract, authentication method, and deployment assumptions.

---

## Step 3: Initial Project Setup

**Instruction file:** `4. Project Initial Setup.md`  
**Input:** PRD + Step 1 planning outputs + Step 2 system design  
**Output:**

```text
project-root/
├── src/                       frontend
├── server/ or api/             backend
├── prisma/ or drizzle/         DB schema and migrations
├── .github/workflows/          CI
├── docker-compose.yml          local development environment
├── .env.example
└── README.md
```

### Completion Criteria

- Frontend passes typecheck, lint, build, and dev-server startup.
- Backend passes typecheck, lint, build, and API-server startup.
- DB migrations run successfully and seed data is created.
- CI runs typecheck, lint, and tests automatically on pull requests.
- Shared UI components match `design-system.md` tokens.
- `.env.example` contains all required environment variables.
- The project uses isolated infrastructure: separate Git repository, DB/project instance, environment variables, deployment settings, and project-specific configuration.

### Before Moving to Step 4

Commit the initialized project:

```bash
git add .
git commit -m "chore: initial project setup"
```

## Step 4: Detailed Development Task Creation

**Instruction file:** `5. Task Creation.md`  
**Input:** PRD + planning outputs + architecture outputs + initialized code patterns  
**Output:**

```text
docs/plans/tasks/
├── {TASK-ID}.md
├── _dependency-graph.md
└── _traceability-matrix.md
```

### Important Rule

Before writing tasks, inspect the code patterns generated in Step 3:

- frontend: shared UI components, API client, Zustand or state-store pattern
- backend: router structure, service layer, repository pattern, error handling
- database: ORM pattern, migration convention, seed pattern

### Completion Criteria

- Cross-validation passes.
- Dependency graph is created.
- Every `F-XX-XX` feature in the specs maps to at least one task.
- Every API endpoint in `api-contract.md` maps to a backend or full-stack task.
- Screen inventory is complete: every `SCR-XX-XX` in wireframes maps to a frontend or full-stack task.
- Traceability matrix is complete: `Spec → API → Task` has no blanks.
- CRUD backend tasks have corresponding frontend management UI tasks, unless explicitly deferred with a reason.

---

## Step 5: Development Execution

**Instruction file:** `6. Development Execution.md`  
**Input:** Step 4 tasks + Step 1–3 outputs  
**Output:** implemented source code, unit tests, API tests, commits

### Full-Stack Execution Strategy

```text
1. Backend tasks first
   DB migration → service logic → API endpoint → API tests

2. Frontend tasks after API completion
   components → state management → API integration → UI validation

3. Full-stack tasks after both foundations are ready
   backend API → frontend component → integration validation
```

### Completion Criteria

- Every task status is `validated/completed`.
- Frontend passes typecheck, lint, test, and build.
- Backend passes typecheck, lint, test, and build.
- Every API endpoint works through `curl`, `httpie`, or equivalent API verification.
- Every policy clause is implemented in code.
- Unit test files exist: at least one `*.test.ts` per backend task and one `*.test.tsx` per frontend task.
- Every frontend task passes wireframe comparison validation.
- Feature-group specification comparison passes across Spec ID, Wireframe, Policy, and role-based flow.
- Visual Verification Gate passes for every screen in the screen inventory.

---

## Step 6: Testing

**Instruction file:** `7. Testing.md`  
**Input:** Step 5 implementation + Step 0–2 documents  
**Output:**

```text
tests/integration/              API + DB + service tests
e2e/                            Playwright E2E tests
tests/security/                 security check results
docs/plans/test-plan.md         test plan
docs/logs/YYYY-MM-DD.md         test result log
```

### Start Condition

Start only after every Step 5 completion criterion is satisfied.

### Testing Phases

1. Integration tests: API endpoint tests, DB integration tests, service-to-service tests.
2. E2E tests: user flows, error codes, state transitions, policies, permissions, and UI behavior.
3. Security checks: dependency vulnerabilities, authentication/authorization validation, input validation.
4. Optional performance tests: only when PRD Section 13.1 specifies performance requirements.

### Completion Criteria

- Integration tests pass.
- E2E tests pass.
- Layout assertion tests pass for wireframe structure, grid, columns, and component types.
- Security check passes with 0 critical/high vulnerabilities.
- Any failure and fix history is recorded in `docs/logs/`.

---

## Step 7: Deployment Preparation

**Instruction file:** `8. Deployment Preparation.md`  
**Input:** test-passed code + PRD + planning + architecture documents  
**Output:** deployment configuration, monitoring settings, rollback plan

### Completion Criteria

- Environment variables are separated by dev, staging, and production.
- DB migration deployment plan and rollback plan are documented.
- Health-check endpoint is verified.
- Monitoring and alerting settings are documented.
- Rollback procedures are written and tested where possible.
- Deployment checklist is complete.

---

## Handoff Rules Between Steps

| Handoff | Condition | Caution |
|---|---|---|
| Step 0 → 1 | PRD self-validation passed + user review | Do not plan before module boundaries and data model are confirmed. |
| Step 1 → 2 | Planning cross-validation passed + user review | Do not design architecture before planning is confirmed. |
| Step 2 → 3 | Architecture reviewed + user approval | Do not initialize the project without confirmed DB/API/auth decisions. |
| Step 3 → 4 | FE+BE typecheck/lint/build passed | Do not create tasks on top of a broken project skeleton. |
| Step 4 → 5 | Task cross-validation passed + dependency graph exists | Do not implement unvalidated tasks. |
| Step 5 → 6 | Full build passed + tasks validated + unit tests exist | Do not start integration/E2E testing without unit tests. |
| Step 6 → 7 | Integration + E2E + security tests passed | Do not prepare deployment from a failing test state. |

## Pipeline Status Tracking

Maintain a project-local status file so progress can resume after the session is interrupted.

```text
docs/PIPELINE-STATUS.md
```

Recommended fields:

```markdown
# Pipeline Status

- Project: {project-name}
- Status: waiting-for-user-review / in-progress / completed / blocked
- Current Step: Step {N} - {name}
- Last Completed Step: Step {N}
- Next Action: {what the user or agent should do next}
```

## Usage Examples

### Full manual pipeline from an idea

```text
1. “Create a PRD for this service idea.”
2. After review: “Create the planning documents.”
3. After review: “Create the system design.”
4. After review: “Initialize the project.”
5. “Create the development tasks.”
6. “Execute development.”
7. “Run testing.”
8. “Prepare deployment.”
```

### Full automatic pipeline after PRD approval

```text
“Run everything.”
```

Use `Auto Execution_EN.md` for the continuous self-validation mode.

### Step-by-step automatic pipeline with review gates

```text
“Run step-by-step.”
```

Use `Step-by-Step Auto Execution_EN.md` for the user-review-gated mode.
