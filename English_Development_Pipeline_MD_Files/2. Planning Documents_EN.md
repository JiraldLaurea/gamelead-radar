# Planning Document Generation Instruction — Step 1

Generate detailed planning documents for every module based on the approved PRD. The planning outputs become the direct source for system design, task generation, development execution, and E2E testing.

---

## Required Work Before Generation

Before creating planning documents, complete the following:

1. Read the entire `PRD.md` from the project root.
   - Pay special attention to Section 2 Module Definitions, Section 3 Permissions, Section 4 Screens, Section 6 Data Model, Section 8 Policies, Section 9 API Endpoint List, and Section 11 Error Codes.
2. Confirm or create the directory:

```text
{project-root}/docs/plans/
```

Do not begin generation before these steps are completed.

---

## Required Output Paths

```text
{project-root}/docs/plans/
├── {module}/
│   └── spec.md
├── policies/
│   └── {policy}.md
├── wireframes/
│   ├── {module}.html
│   ├── Storyboard.html                  optional
│   └── screens/                         optional
│       └── {flowId}_{stepNumber}_{screenName}.png
├── design-system.md
└── ui-implementation-rules.md
```

### Path Rules

- `{module}` must use the PRD module ID in kebab-case.
- `{policy}` must use the policy domain in kebab-case.
- The module name used in `spec.md` and `wireframes/{module}.html` must match.
- These paths are referenced exactly by task creation, development execution, and E2E testing. Do not rename them without updating all downstream references.

---

## Output Set Per Module

Each UI module normally requires:

1. `docs/plans/{module}/spec.md`
2. `docs/plans/policies/{policy}.md` if the module has policy rules
3. `docs/plans/wireframes/{module}.html`

Backend-only or internal modules may omit the wireframe, but `spec.md` must clearly state:

```markdown
wireframe: N/A (no UI screen — internal service module)
```

# 1. `spec.md` — Functional Specification

Each module specification must include:

```markdown
# {Module Name} Specification

## 1. Overview
## 2. Module Positioning
## 3. Roles and Permissions
## 4. Data Model
## 5. Functional Requirements
## 6. User Flows
## 7. Screen List
## 8. Edge Cases
## 9. API Integration Mapping
## 10. Data Flow
## 11. Policy References
## 12. Test Sources
```

### Required Details

- All entities used by the module, with fields, types, and enum values.
- Every feature ID from the PRD that belongs to this module.
- Every screen ID mapped to the corresponding wireframe section ID.
- Edge cases mapped to error codes.
- API mapping based only on PRD Section 9. Do not invent endpoints.

### API Integration Mapping

```markdown
## API Integration Mapping

| Feature ID | Feature Name | API Endpoint | Method | Source |
|---|---|---|---|---|
| F-AUTH-01 | Login | /api/auth/login | POST | PRD Section 9 |
| F-AUTH-02 | Signup | /api/auth/register | POST | PRD Section 9 |
| F-AUTH-03 | Get Profile | /api/users/me | GET | PRD Section 9 |
```

Only reference endpoints defined in PRD Section 9.

### Data Flow

Write the full frontend-to-backend-to-database flow for every feature.

```markdown
## Data Flow

### F-AUTH-01: Login
1. FE: LoginForm → Zod validation → useMutation(POST /api/auth/login)
2. API: /api/auth/login → AuthController.login()
3. Service: AuthService.login() → password verification → token/session creation
4. DB: SELECT users WHERE email = ? → INSERT sessions
5. Response: { accessToken, refreshToken, user } → FE TokenStore → redirect
```

Do not omit simple GET flows.

# 2. `policy.md` — Policy Document

Each policy document must include:

```markdown
# {Policy Domain} Policy

## 1. Purpose
## 2. Policy Scope
## 3. Policy Rules
## 4. Frontend Validation
## 5. Backend Validation
## 6. DB Constraints
## 7. Error Code Mapping
## 8. Test Requirements
```

### Policy Rule Format

```markdown
## AUTH-P-01: Password must be at least 8 characters and include letters, numbers, and special characters

| Validation Layer | Validation Method | Behavior on Violation |
|---|---|---|
| FE | Zod schema: `z.string().min(8).regex(...)` | Show inline error message |
| BE | Same Zod rule + API validation middleware | 400 Bad Request `{ code: "INVALID_PASSWORD", message: "..." }` |
| DB | CHECK constraint or application-level validation | Reject invalid data |
```

Frontend validation is for UX. Backend validation is the actual security boundary.

### Policy → DB Constraint Mapping

```markdown
| Policy ID | Policy Content | DB Constraint | Table.Column |
|---|---|---|---|
| AUTH-P-01 | Minimum password length | Application-level after hashing | users.password_hash |
| AUTH-P-05 | Unique email | UNIQUE constraint | users.email |
| BOOK-P-03 | Max 10 bookings per user per day | Application-level + COUNT query | bookings(user_id, date) |
```

# 3. `wireframe.html` — Interactive Wireframe

Create one interactive HTML wireframe per UI module.

### Module-Wireframe Comparison Rule

- Every PRD UI module must have one `wireframes/{module}.html`.
- Backend-only modules may be marked N/A in `spec.md`.
- Cross-validation must confirm:

```text
number of modules = number of wireframes + number of N/A internal modules
```

### Section ID Rules

- Every screen or major section must have a unique ID using:

```html
<section id="SCR-{MODULE}-{number}">
```

- Role-specific screens must use separate screen IDs.
  - Owner dashboard: `SCR-TASK-01`
  - Manager dashboard: `SCR-TASK-02`
- Use tokens from `design-system.md`.
- Include state-specific UI: default, active, error, loading, empty, disabled, and success states.
- Preserve real text labels that should later appear in the UI.

# 5. `ui-implementation-rules.md`

Create `docs/plans/ui-implementation-rules.md` for UI projects.

### Required Contents

- How wireframe sections map to actual React components.
- How `SCR-...` IDs must be preserved or referenced in implementation.
- How loading, error, empty, and disabled states must be implemented.
- How design-system tokens must be reflected in Tailwind config or CSS variables.
- How accessibility rules are applied: labels, focus states, keyboard navigation, semantic elements.
- How visual verification screenshots must be captured and compared.
- How role-based screens must be rendered and tested.

If the file already exists, check consistency with the latest `design-system.md`.

---

## Core Consistency Rules

- Every `F-XX-XX` feature in `spec.md` must map to a screen section ID or `internal`.
- Every screen ID in the PRD must appear in the wireframe or be explicitly marked as deferred.
- Shared enums must reference the owning module defined in PRD Section 6.3.
- Do not redefine the same enum differently in multiple module specs.
- Every policy rule must be reflected in either UI behavior, backend validation, or both.
- Numeric policy values must match across PRD, spec, policy, and wireframe.
- API integration mappings must match PRD Section 9 exactly.
- Policy server-side validation must map to PRD Section 11 error codes.

---

## Cross-Validation Checklist

### PRD ↔ Spec

- Every PRD module has a module spec.
- Every PRD feature appears in exactly one primary module spec.
- Every feature keeps the same ID and priority.
- Every feature maps to a screen ID or `internal`.

### Spec ↔ Wireframe

- Every screen in the spec exists in the wireframe.
- Every wireframe section ID exists in the spec screen list.
- Role-specific screens are separated.
- Every important UI state is represented.

### Spec ↔ API

- Every feature requiring an API maps to a PRD-defined endpoint.
- No endpoint is invented in the spec.
- API method and path are exactly the same as the PRD.

### Policy ↔ Error Codes

- Every policy rule has a violation behavior.
- Every server-side violation maps to an error code.
- Error code names match PRD Section 11.

### Design System ↔ Wireframe

- Wireframes use the defined color, spacing, component, and typography tokens.
- Status colors match enum/status definitions.
