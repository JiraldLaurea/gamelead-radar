# Detailed Development Task Creation Instruction — Step 4

Create detailed development task documents based on the PRD, planning outputs, system design, and initialized project code patterns. The goal is to eliminate missing implementation work before coding begins.

---

## Required Work Before Task Creation

Read all of the following before creating tasks:

1. `PRD.md`
2. All files under `docs/architecture/`
   - `architecture.md`
   - `db-design.md`
   - `api-contract.md`
3. All planning files under `docs/plans/`
   - every module `spec.md`
   - every policy document
   - every wireframe HTML
   - `design-system.md`
   - `ui-implementation-rules.md`
4. Existing code structure from Step 3
   - `src/`
   - `server/` or `api/`
   - `prisma/` or `drizzle/`
   - shared components
   - API client
   - validation schemas
   - existing test patterns

Do not create tasks until these are inspected.

---

## Required Outputs

Create task files inside the project directory:

```text
docs/plans/tasks/
├── {TASK-ID}.md
├── _dependency-graph.md
├── _traceability-matrix.md
├── _screen-inventory.md
└── _task-summary.md
```

# Required Pre-Task Deliverable 1: Traceability Matrix

Before writing task files, create a complete `Spec Feature → API → Task` traceability matrix.

```markdown
# Traceability Matrix

## {Module Name}

| Spec ID | Feature Name | Required API Endpoint | Exists in api-contract.md | BE Task | FE Task | Notes |
|---|---|---|---|---|---|---|
| F-AUTH-01 | Login | POST /api/auth/login | ✅ | BE-AUTH-1 | FE-AUTH-1 | |
| F-AUTH-02 | Signup | POST /api/auth/register | ✅ | BE-AUTH-1 | FE-AUTH-2 | |
| F-AUTH-03 | Profile | GET /api/users/me | ✅ | BE-AUTH-2 | FE-AUTH-3 | |
| F-UI-01 | Toggle tab | FE only | N/A | N/A | FE-UI-1 | UI state only |
```

## Rules

1. List every `F-XX-XX` feature from every spec.
2. Identify the required API endpoint for each feature.
3. Verify the endpoint exists in `api-contract.md`.
4. If the endpoint is missing, update `api-contract.md` before creating tasks or mark the feature as intentionally excluded with a reason.
5. Map every backend API to a backend or full-stack task.
6. Map every screen/UI feature to a frontend or full-stack task.
7. Empty cells are treated as expected implementation omissions and must be resolved.

## Validation

- Every feature row is complete.
- Every required API endpoint exists or is explicitly deferred.
- Every API maps to at least one BE/FS task.
- Every screen maps to at least one FE/FS task.
- Excluded items include a clear reason.

# Required Pre-Task Deliverable 2: BE-FE CRUD Correspondence Check

Every backend CRUD API must have a corresponding frontend management UI task unless explicitly excluded.

```markdown
# BE-FE CRUD Correspondence

| BE Task | CRUD API | FE Task | Management UI Location | Wireframe Screen | Status |
|---|---|---|---|---|---|
| BE-CATEG-1 | /api/categories CRUD | FE-CATEG-1 | /settings/categories | SCR-SET-03 | ✅ |
| BE-USERS-1 | /api/users CRUD | FE-USERS-1 | /settings/users | SCR-USERS-02 | ✅ |
```

## Rules

- Extract all backend tasks that create, update, or delete records.
- Confirm that users have a UI to manage those records.
- If no UI exists, add a frontend task or state that it is intentionally excluded.
- Do not complete task creation with a blank FE task column for CRUD APIs.

# Required Pre-Task Deliverable 4: FE-BE Field Naming Strategy

Decide and document the field naming strategy before writing tasks.

## Recommended Strategy A: Unified camelCase

Use camelCase across frontend, backend, validation schemas, and API contracts. Recommended for TypeScript full-stack projects.

```text
Frontend request: assigneeId
Backend Zod schema: assigneeId
Database ORM field: assigneeId or mapped to assignee_id by ORM
```

## Strategy B: API camelCase + DB snake_case

Use camelCase at the API boundary and let ORM mapping handle DB snake_case.

## Strategy C: Explicit Conversion Layer

Use conversion functions when backend APIs use snake_case and frontend uses camelCase.

```typescript
function toApiTask(input: TaskFormValues): CreateTaskRequest {
  return {
    assignee_id: input.assigneeId,
  };
}
```

## Required Common Task

If conversion is needed, create a common task:

```text
SHARED-FIELD-1: Implement API field-name conversion utilities and tests
```

# Task Decomposition Rules

## Backend Tasks

Backend tasks should be organized around stable units:

- DB migration/schema
- repository
- service logic
- controller/API route
- validation schema
- error handling
- unit/API tests

Do not combine unrelated domains in one backend task.

## Frontend Tasks

Frontend tasks should be organized around screens or flows:

- route/page
- components
- form validation
- API integration
- state management
- loading/error/empty states
- visual verification
- component tests

Every frontend task must reference the wireframe section IDs it implements.

## Full-Stack Tasks

Use full-stack tasks only when frontend and backend changes are tightly coupled and cannot be safely separated.

## Infrastructure Tasks

Infrastructure tasks include:

- Docker setup
- CI setup
- environment variable setup
- DB initialization
- shared type generation
- logging/monitoring setup

# Dependency Graph

Create `_dependency-graph.md`.

```markdown
# Dependency Graph

## Execution Groups

### Group 1 — Infrastructure
- INF-DB-1
- INF-CI-1

### Group 2 — Backend Foundation
- BE-AUTH-1 depends on INF-DB-1
- BE-USERS-1 depends on BE-AUTH-1

### Group 3 — Frontend Foundation
- FE-AUTH-1 depends on BE-AUTH-1

### Group 4 — Feature Integration
- FS-BOOKING-1 depends on BE-BOOKING-1 and FE-BOOKING-1
```

## Rules

- Backend DB tasks must run before API tasks that depend on them.
- Frontend API integration tasks must run after corresponding backend APIs are implemented.
- Shared type tasks must run before FE/BE tasks that import those types.
- E2E test tasks must run after the user flow is implemented.

# Validation Checklist

## Traceability

- Every feature maps to at least one task.
- Every API endpoint maps to at least one backend/full-stack task.
- Every wireframe screen maps to at least one frontend/full-stack task.
- Every policy maps to implementation and test tasks.

## BE-FE Correspondence

- Every CRUD API has a corresponding UI or explicit exclusion.
- Every FE API call has a completed or planned BE endpoint.
- Field naming strategy is documented.

## Test Coverage

- Backend tasks include unit or API tests.
- Frontend tasks include component or behavior tests.
- Full-stack tasks include integration or E2E test references.
- Security-sensitive tasks include security test requirements.

## Code Pattern Alignment

- Tasks reference existing project patterns.
- Tasks avoid introducing duplicate API clients, duplicate stores, or duplicate validators.
- Tasks identify exact files to create or modify.
