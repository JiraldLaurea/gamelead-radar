# Stage 1: Environment-Specific Configuration Validation

## 1-1. Environment Variables

Create a table from `.env.example`.

```markdown
| Variable | Description | dev | staging | prod | Sensitive |
|---|---|---|---|---|---|
| DATABASE_URL | DB connection | docker-compose | staging DB | prod DB | ✅ |
| JWT_SECRET | auth secret | local secret | secret manager | secret manager | ✅ |
| NODE_ENV | runtime mode | development | staging | production | ❌ |
| EXTERNAL_API_KEY | external service | sandbox | sandbox | production | ✅ |
```

### Validation

- `.env.example` includes every required variable.
- Sensitive values are not hardcoded.
- Production secrets will be stored in a secret manager or secure environment variable system.
- External service keys are separated by environment.
- Local/dev values cannot accidentally point to production resources.

# Stage 2: DB Migration Deployment Plan

## 2-1. Check Migration Status

```bash
pnpm exec prisma migrate status
# or
pnpm exec drizzle-kit check
```

## 2-2. Migration Execution Order

```markdown
| Order | Migration | Change | Rollback Possible | Caution |
|---|---|---|---|---|
| 1 | 20260601_init | create base tables | yes | low risk |
| 2 | 20260602_add_index | add index | yes | run during low traffic |
```

## 2-3. Rollback Feasibility

For each migration, confirm:

- down migration or rollback SQL exists
- rollback does not cause unacceptable data loss
- application still works after rollback
- backups exist before destructive operations

Local rollback test:

```bash
pnpm exec prisma migrate reset --skip-seed
pnpm exec prisma migrate dev
pnpm exec prisma db seed
```

## 2-4. Large Table Cautions

| Operation | Risk | Response |
|---|---|---|
| Add index | medium | use concurrent index creation where supported |
| Add NOT NULL column with default | medium | handle existing rows first |
| Delete column | high | deploy code first, remove column later |
| Change column type | high | add new column → copy data → remove old column |
| Delete table | high | rename/archive before permanent removal |

## 2-5. Data Loss Review

- Is any column being deleted?
- Is any table being deleted?
- Is an enum value being removed?
- Is a NOT NULL constraint being added?
- Is existing nullable data handled?
- Is backup/restore tested?

# Stage 3: Dockerfile and Build Verification

## Recommended Multi-Stage Dockerfile

```dockerfile
FROM node:20-alpine AS deps
WORKDIR /app
COPY package.json pnpm-lock.yaml ./
RUN corepack enable && pnpm install --frozen-lockfile

FROM node:20-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN corepack enable && pnpm build

FROM node:20-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
COPY --from=builder /app/.next ./.next
COPY --from=builder /app/public ./public
COPY --from=builder /app/package.json ./package.json
COPY --from=deps /app/node_modules ./node_modules
EXPOSE 3000
CMD ["pnpm", "start"]
```

## Docker Checks

- `.dockerignore` includes `node_modules`, `.next`, `.env*`, test results, and local files.
- Final image excludes dev-only files when possible.
- Build-time and runtime environment variables are separated.
- Health endpoint responds inside the container.
- DB migration execution timing is defined: entrypoint, CI job, or manual release step.

## Apple Silicon → AMD64 Caution

If deploying to AMD64 infrastructure from Apple Silicon, specify platform:

```bash
docker build --platform linux/amd64 -t {image-name}:latest .
```

- health endpoint works
- login/auth works
- primary user flow works
- CRUD create/update/delete works for core entities
- permission restrictions work
- external service integration works or degrades gracefully
- logs are being collected
- alerts are active
