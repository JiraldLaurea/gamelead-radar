# Testing Instruction — Step 6

Perform integration testing, E2E testing, security checks, and optional performance testing after development is complete. Testing must be orchestrated so that code correctness and actual UI behavior are both verified.

---

## Core Principle

CLI E2E success alone does not guarantee that the screen is visually correct. Playwright tests must be combined with actual screen verification where possible.

Use an orchestrator-subagent structure when working in Claude Code:

```text
Orchestrator
├── Subagent A: CLI test runner
├── Subagent B: Playwright/MCP screen verification
└── Subagent C: Test healer for failed tests
```

## Testing Strategy

```text

  API + DB + services
  ↓
Phase 2: E2E Tests
  user flows + screens + policies + permissions
  ↓
Phase 3: Security Checks
  vulnerabilities + auth + validation
  ↓
```

Follow the order. If integration tests fail, do not proceed to E2E.

## API Endpoint Test Rule

Each endpoint should include:

- success case
- invalid input case
- unauthorized case if auth required
- forbidden case if role-restricted
- not found case if resource-based
- conflict case if uniqueness/state conflict exists
- DB persistence check for create/update/delete

Example:

```typescript
describe('POST /api/auth/login', () => {
  it('logs in with valid credentials', async () => {
    // arrange
    // act
    // assert response
    // assert session/token behavior
  });
});
```

## Integration Validation Checklist

- Every endpoint in `api-contract.md` is tested.
- Every server-side error code is tested.
- Every allowed/forbidden state transition is tested.
- Every role-feature permission combination is tested.
- Every policy server-side validation rule is tested.
- FE-BE payload keys are compatible with backend Zod schemas.

## E2E Validation Criteria

### UI Rendering

- Required elements exist.
- Text labels are correct.
- Buttons and inputs are visible and enabled/disabled correctly.
- Loading, error, empty, success, and disabled states are shown correctly.
- Role-specific UI branches render correctly.

### CRUD Data Flow

For create/update/delete features, verify real data behavior:

1. Create data through UI.
2. Confirm it appears in the UI.
3. Confirm it exists through API or DB query if possible.
4. Update the data.
5. Confirm the updated value persists after refresh.
6. Delete or archive the data.
7. Confirm it disappears or changes state correctly.

This prevents silent bugs such as FE sending `assigneeId` while BE expects `assignee_id`.

### Policy and Permission Behavior

- Unauthorized users are redirected or blocked.
- Forbidden roles see the correct error or access-denied screen.
- Policy violations show the correct user-facing message.
- Forbidden state transitions are rejected.


---

# Test Healing Protocol

When E2E tests fail:

1. Read the failure output.
2. Open the actual page with Playwright/MCP if available.
3. Capture the browser snapshot.
4. Identify whether the issue is test-selector drift, UI bug, API bug, seed data issue, or timing issue.
5. Fix the correct layer.
6. Re-run the failed test.
7. Re-run the full E2E suite if shared behavior changed.

Do not blindly update tests to match broken UI.

# Phase 3: Security Checks

## Required Checks

### Dependency Vulnerabilities

```bash
pnpm audit
```

### Authentication

- protected endpoints reject unauthenticated requests
- invalid tokens are rejected
- expired tokens are rejected
- logout invalidates session/token if applicable

### Authorization

- each role is blocked from forbidden actions
- own-data-only restrictions are enforced server-side
- admin-only APIs cannot be accessed by normal users

### Input Validation

- invalid body rejected
- invalid query parameters rejected
- file upload limits enforced if applicable
- injection-like input is handled safely

### Secrets

- `.env` is not committed
- `.env.example` has no real secrets
- production secrets are not hardcoded

### Rate Limiting / Abuse Prevention

If required by architecture:

- rate limit triggers correctly
- error response matches API contract
