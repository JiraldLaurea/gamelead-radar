# PRD Writing Instruction

Write a complete Product Requirements Document based on the user's idea, requirements, and reference materials. The PRD becomes the Single Source of Truth for downstream planning, task creation, development execution, and E2E testing.

---

## Confirm Before Writing the PRD

If the following information is missing, ask the user before writing the PRD:

1. Project type: web app, mobile app, API service, admin tool, full-stack product, etc.
2. User roles: number and type of roles. If there is only one role, simplify the permission matrix.
3. Preferred technology stack: use a specified stack if already decided; otherwise recommend one in the PRD.
4. MVP scope: whether to build everything at once or split into phases.
5. External services: payment gateway, authentication provider, cloud provider, messaging service, analytics service, etc.

---

## Required PRD Sections

The PRD must include the following 16 sections. If a section is not applicable, write `N/A` and explain why. Do not leave any section blank.

```markdown
1. Project Overview
2. Module Definitions
3. User Roles and Permission Matrix
4. Screen/Page/View List
5. Module-Specific Functional Requirements
6. Data Model
7. State Transition Rules
8. Business Logic and Policies
9. API Endpoint List
10. External Service Integrations
11. Error Codes and Messages
12. Notification/Event System
13. Non-Functional Requirements
14. MVP / Phase Definition
15. Technology Stack
16. Glossary
```

## 1. Project Overview

```markdown
### 1. Project Overview
- **Service Name**:
- **Core Concept**: 1–2 sentences explaining what the service is.
- **Project Type**: web app / mobile app / API service / admin tool / full-stack.
- **Primary Target Users**: who uses the service.
- **Service Scope**: region, market, domain.
- **Core Value Proposition**: 1–3 reasons users would use the service.
```

## 2. Module Definitions

Planning documents are generated per module, so module boundaries must be finalized in the PRD.

```markdown
### 2. Module Definitions

| Module ID | Module Name | Description | Included Features | Dependent Modules |
|---|---|---|---|---|
| {kebab-case} | {name} | {one-line description} | {major features} | {module ID or "-"} |
```

### Rules

- Module IDs must use kebab-case because they become file paths such as `docs/plans/{module-id}/spec.md`.
- Dependent modules must be specified so planning and task generation can determine order.
- One module generally creates one `spec.md`, one `wireframe.html`, and related policy documents.
- Separate modules by independently deployable unit or domain boundary.

### Module Separation Guide

- Authentication/user management should usually be a separate module.
- Different CRUD targets should usually be separate modules.
- Role-specific views for the same CRUD target should usually remain in the same module.
- Real-time or asynchronous processing, such as notification or chat, should usually be a separate module.

## 3. User Roles and Permission Matrix

The permission matrix is the source for access-control policy documents.

```markdown
### 3. User Roles and Permission Matrix

#### 3.1 Role Definitions
| Role | Description | Signup/Creation Condition |
|---|---|---|
| {ROLE_NAME} | {description} | {creation method} |
```

Even if the project has only one role, write it explicitly. If unauthenticated users have features, include `GUEST` as a role.

```markdown
#### 3.2 Feature-Level Permissions
| Feature | {ROLE_1} | {ROLE_2} | {ROLE_N} |
|---|---|---|---|
| {feature name} | {permission} | {permission} | {permission} |
```

### Permission Notation

- `✅` Allowed
- `❌` Forbidden
- `🔒` Own data only
- `👀` Read-only
- Conditional: `✅ (after approval)`

### Rules

- Every PRD feature must appear in the matrix.
- This matrix becomes the source for role-based access policy rules.

---

## 4. Screen/Page/View List

Define all screens before wireframe generation. For API-only projects with no UI, write `N/A`.

```markdown
### 4. Screen List

| Screen ID | Screen Name | URL/Path | Allowed Roles | Module | Description |
|---|---|---|---|---|---|
| SCR-{MODULE}-{number} | {screen name} | {URL pattern} | {roles} | {module ID} | {one-line description} |
```

### Rules

- Screen IDs must follow `SCR-{MODULE}-{number}`. Numbers start from `01`.
- Every screen must specify allowed roles.
- This list must map 1:1 with wireframe screens.
- Include modals, bottom sheets, drawers, and major panels as separate rows when they represent distinct user interactions.
- Use `[id]` or `[slug]` for dynamic URL segments.

## 5. Module-Specific Functional Requirements

Each module must include features, user flows, and edge cases.

```markdown
### 5.N {module-id} Module

#### Feature List
| Feature ID | Feature Name | Screen | Role | Priority | Description |
|---|---|---|---|---|---|
| F-{MOD}-{number} | {feature name} | {SCR-ID or "internal"} | {role} | P0/P1/P2 | {detailed description} |

#### User Flows
| Flow ID | Flow Name | Steps | Success Condition |
|---|---|---|---|
| UF-{MOD}-{number} | {flow name} | {step-by-step flow} | {success condition} |

#### Edge Cases
- {situation} → {error code E-{MODULE}-{number}} or {fallback behavior}
```

### Rules

- Feature IDs must be unique within each module.
- Every feature must map to a screen ID. Backend-only features should use `internal`.
- User flow IDs become the source for E2E test cases.
- Edge cases should reference error codes from Section 11.
- Priority definitions: `P0 = MVP required`, `P1 = important`, `P2 = nice to have`.

## 6. Data Model

```markdown
### 6. Data Model

#### 6.1 Entity Relationship Diagram
Write an ERD using Mermaid, ASCII, or a clear table-based relationship diagram.

#### 6.2 Entity Definitions
| Field | Type | Required | Description |
|---|---|---|---|
| {field} | {type} | O / - | {description; if FK, write "FK → Entity"} |

#### 6.3 Enum Definitions
| Enum Name | Owning Module | Value | Label | Description |
|---|---|---|---|---|
| {EnumName} | {module} | {VALUE} | {label} | {meaning} |
```

### Rules

- Every entity must include primary key, timestamps, ownership fields, and status fields when applicable.
- Every foreign key must state its target entity.
- Every enum must define allowed values and labels.
- If multiple modules share an enum, specify one owning module and reference it from other modules.

## 7. State Transition Rules

Every stateful entity must have a transition table.

```markdown
### 7. State Transition Rules

#### {Entity}.{statusField}
| Current State | Event/Action | Next State | Allowed Roles | Policy/Error Code |
|---|---|---|---|---|
| {CURRENT} | {action} | {NEXT} | {roles} | {policy or error code} |
```

### Rules

- Define allowed transitions and forbidden transitions.
- Forbidden transitions must map to error codes.
- These rules become sources for integration and E2E tests.

---

## 8. Business Logic and Policies

```markdown
### 8. Business Logic and Policies

| Policy ID | Policy Name | Rule | Applies To | Error Code |
|---|---|---|---|---|
| {MOD}-P-{number} | {policy name} | {specific numeric/conditional rule} | {feature/API/entity} | E-{MOD}-{number} |
```

### Rules

- Avoid vague expressions such as “after some time” or “as needed.” Use specific values such as “after 10 minutes.”
- Every policy must specify frontend behavior, backend validation, and error handling.
- Every policy must map to one or more error codes.

## 9. API Endpoint List

```markdown
### 9. API Endpoint List

| Endpoint ID | Method | Path | Module | Auth Required | Request | Response | Error Codes |
|---|---|---|---|---|---|---|---|
| API-{MOD}-{number} | POST | /api/{resource} | {module} | yes/no | {request summary} | {response summary} | {errors} |
```

### Rules

- Every feature requiring backend communication must map to an API endpoint.
- Do not define endpoints that cannot be mapped to a feature or internal service requirement.
- API paths and methods must later match `api-contract.md` exactly.

## 11. Error Codes and Messages

```markdown
### 11. Error Codes and Messages

| Error Code | HTTP Status | Message | Trigger Condition | User-Facing Message |
|---|---|---|---|---|
| E-{MOD}-{number} | 400/401/403/404/409/500 | {developer message} | {condition} | {message shown to user} |
```

### Rules

- Every policy violation must have an error code.
- Every forbidden state transition must have an error code.
- Error codes must be stable and should not be renamed without migration notes.

## PRD Self-Validation Checklist

### Completeness

- All 16 sections exist.
- No section is blank.
- `N/A` sections include a reason.
- Every feature has a module, role, priority, and screen/internal mapping.

### Consistency

- Module IDs are consistent across all sections.
- Feature IDs are unique.
- Screen IDs are unique.
- Every policy maps to an error code.
- Every enum has an owning module and allowed values.
- Every stateful entity has state-transition rules.

### Specificity

- Policy thresholds are numeric and explicit.
- Non-functional requirements include measurable targets where possible.
- API endpoints include method, path, request, response, and errors.

### Downstream Compatibility

- The PRD can generate planning documents without guessing.
- The PRD can generate architecture documents without guessing.
- The PRD can generate tasks without missing API/screen mappings.
- Deployment assumptions are clear enough for Step 7.

## Output Rule


```markdown
## PRD Ready for Review

- PRD path: PRD.md
- Self-validation: passed / failed
- Key review points:
  - module boundaries
  - data model
  - policy thresholds
  - API endpoint list
  - error codes
  - deployment assumptions
```
